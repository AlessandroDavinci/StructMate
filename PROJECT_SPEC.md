# “智构学伴 StructMate”项目总体方案

## 一、项目名称

**中文名称：智构学伴**

**英文名称：StructMate**

**项目全称：基于大模型多智能体协同的数据结构个性化学习资源生成系统**

**项目定位语：**

面向数据结构零基础本科生，通过动态学习画像、多智能体资源生成和个性化学习路径规划，帮助学生解决“不知道学什么、看不懂、不会练、不会写”的学习问题。

---

## 二、项目核心定位

智构学伴不是普通的大模型问答工具，而是一套面向数据结构课程的“学习决策与资源生成系统”。

系统围绕以下闭环运行：

学生对话输入
→ 自动构建学习画像
→ 诊断当前知识基础
→ 规划个性化学习路径
→ 多智能体协同生成学习资源
→ 完成练习并分析错误
→ 更新画像与后续学习建议

项目重点从“回答学生问题”转变为“帮助学生决定如何学习，并自动准备适合其当前水平的资源”。

---

## 三、目标用户

### 1. 核心用户

计算机相关专业本科低年级学生，尤其是：

* 尚未系统学习数据结构的学生；
* C/C++或其他编程语言基础较弱的学生；
* 对抽象数据结构概念理解困难的学生；
* 不知道课程知识点学习顺序的学生；
* 能理解理论但无法独立完成代码实现的学生；
* 面对大量网络资源无法进行筛选的学生。

### 2. 典型用户画像

计算机专业大一学生，第一次接触数据结构，每天可投入约1小时，希望掌握课程基础并通过期末考试，但对指针、递归、复杂度分析等前置知识掌握不足。

---

## 四、核心用户痛点

### 1. 缺乏明确学习起点

学生不知道应先学习复杂度、数组、指针还是链表，也不了解知识点之间的依赖关系。

### 2. 学习基础差异明显

统一教材和统一教学进度不能同时适应零基础学生与具有编程经验的学生。

### 3. 教材内容抽象

链表、递归、树、图等内容依赖抽象思维，单纯文字和公式难以形成直观理解。

### 4. 练习与反馈不足

学生完成题目后，只能看到正确答案，难以判断错误属于概念理解、代码语法、边界处理还是算法设计问题。

### 5. 理论与代码实践脱节

学生能够复述定义，却无法将抽象数据结构转化为具体代码。

### 6. 学习资源繁杂

网络课程、博客、题库和代码示例数量庞大，但难度和质量差异明显，学生无法判断资源是否适合自身水平。

### 7. 编程错误难以定位

编译错误、空指针、数组越界、递归终止条件错误等问题容易阻断学习过程。

## 五、项目建设目标

### 1. 自动构建动态学生画像

通过自然语言对话获取学生的专业、年级、基础、目标、学习时间和学习偏好，避免填写复杂表单。

### 2. 生成整门课程的个性化学习路径

根据学生基础和目标，从数据结构完整课程知识图谱中选择学习模块，确定学习顺序、阶段目标和预计学习时间。

### 3. 协同生成七类学习资源

系统根据学生画像和当前知识点，生成：

1. 个性化知识讲解；
2. 知识点思维导图；
3. 分层练习题；
4. 错题解析；
5. 代码实操案例；
6. 项目式学习任务；
7. 学习总结卡片。

### 4. 实现轻量学习诊断

分析学生练习结果和错误类型，给出补弱建议，并更新易错知识点、认知水平和学习进度。

### 5. 保证内容安全与基本可靠性

通过结构化知识库、提示词约束、内容审核智能体和代码规则检查降低幻觉风险。

## 六、课程内容范围

系统覆盖一门完整的数据结构课程，按照九个模块组织。

| 模块   | 主要内容            |
| ---- | --------------- |
| 第一模块 | 数据结构基本概念、算法与复杂度 |
| 第二模块 | 线性表、顺序表与链表      |
| 第三模块 | 栈、队列及其应用        |
| 第四模块 | 串、数组与广义表        |
| 第五模块 | 树、二叉树与遍历        |
| 第六模块 | 图、图遍历与最短路径      |
| 第七模块 | 查找与散列表          |
| 第八模块 | 内部排序算法          |
| 第九模块 | 综合课程项目与复习       |

“整门课程”不等于预先生成全部教学资源。系统首先建立完整课程知识结构，再根据学生学习进度按需生成资源，从而降低存储和生成成本。

代码示例默认采用**C++17**，并在配置文件中预留C语言或Python切换选项。

## 七、学生画像设计

系统建立九维动态学习画像。

| 画像维度   | 主要内容           | 获取方式      |
| ------ | -------------- | --------- |
| 专业与年级  | 计算机专业、年级信息     | 初始对话      |
| 知识基础   | 编程、数学、算法基础     | 对话与诊断题    |
| 学习目标   | 通过考试、提高成绩、项目实践 | 初始对话      |
| 认知水平   | 记忆、理解、应用三个层级   | 题目表现      |
| 学习节奏   | 慢速、常规、快速       | 用户选择与学习记录 |
| 内容形式偏好 | 文字、图示、代码、练习    | 初始对话与反馈   |
| 易错知识点  | 指针、递归、复杂度等     | 错题分析      |
| 学习投入度  | 每日学习时间、完成率     | 对话与学习记录   |
| 历史成绩   | 编程课、离散数学等成绩    | 用户自述      |

### 画像动态更新规则

画像不是一次性生成，而是在以下事件发生后进行更新：

* 完成诊断测试；
* 完成章节练习；
* 多次出现同类错误；
* 修改学习目标；
* 学习进度落后或提前；
* 主动反馈资源难度过高或过低。

初赛版本只需实现基于规则的更新，不需要训练机器学习模型。

## 八、系统总体架构

采用“1＋7”多智能体架构。

### 1. 中央协调器

**学习任务协调器 Learning Coordinator**

中央协调器负责：

* 接收用户学习请求；
* 读取学生画像；
* 拆分资源生成任务；
* 决定调用哪些智能体；
* 管理智能体之间的协商；
* 汇总最终结果；
* 保存学习记录。

协调器负责流程控制，但不直接生成具体教学内容。

### 2. 七个专业智能体

#### （1）知识检索智能体

负责从数据结构课程知识库中定位相关章节、知识点、先修关系、重点难点和标准概念。

输入：

* 当前学习主题；
* 学生画像；
* 课程章节信息。

输出：

* 相关知识点；
* 前置知识；
* 核心概念；
* 常见错误；
* 可调用的课程材料。

初赛不实现复杂向量RAG，采用章节标签、关键词和知识点映射表完成检索。

#### （2）教学设计智能体

根据学生基础和目标确定教学深度、讲解顺序、案例类型和练习难度。

例如，对零基础学生采用：

生活类比
→ 图示理解
→ 伪代码
→ 简单代码
→ 基础练习

对基础较好的学生采用：

抽象定义
→ 复杂度分析
→ 完整实现
→ 算法比较
→ 综合任务

#### （3）文档与实践资源生成智能体

负责生成：

* 个性化知识讲解；
* 思维导图文本或Mermaid代码；
* 代码示例；
* 项目式学习任务；
* 学习总结卡片。

该智能体同时承担代码实操资源生成，避免增加过多智能体导致系统复杂。

#### （4）题库生成智能体

根据知识点、画像和学习目标生成分层练习题。

题目分为三个等级：

* 基础层：概念判断、选择和简单填空；
* 应用层：代码阅读、代码补全和结果分析；
* 提升层：算法设计、复杂度分析和综合编程。

每道题包含：

* 题目；
* 难度；
* 考查知识点；
* 标准答案；
* 解析；
* 常见错误；
* 评分标准。

#### （5）学习路径规划智能体

根据课程知识依赖关系和学生画像生成学习路径。

输出内容包括：

* 当前阶段目标；
* 推荐学习顺序；
* 每个模块预计时间；
* 每日或每周任务；
* 配套资源；
* 阶段检查点；
* 学习困难时的补救路径。

例如，若学生指针基础较弱，系统不会直接推荐链表，而是先插入“指针与动态内存基础”补充任务。

#### （6）评估诊断智能体

初赛版本不开发完整的学习效果评估平台，只实现轻量诊断功能：

* 判断答案是否正确；
* 识别错误类型；
* 判断知识掌握程度；
* 更新易错知识点；
* 推荐下一步练习；
* 调整资源难度。

错误类型包括：

* 概念理解错误；
* 算法步骤错误；
* 语法错误；
* 边界条件错误；
* 空指针错误；
* 数组越界；
* 复杂度判断错误；
* 递归终止条件错误。

#### （7）内容审核与防幻觉智能体

负责审核其他智能体生成的结果。

检查内容包括：

* 是否符合数据结构基本概念；
* 是否存在前后矛盾；
* 代码与文字解释是否一致；
* 题目是否存在多个答案；
* 答案与解析是否匹配；
* 是否超出课程知识库范围；
* 是否包含提示词攻击或不安全内容；
* 是否泄露用户隐私数据。

审核不通过时，将修改意见返回对应生成智能体，最多重新生成一次，防止循环调用和响应时间失控。

## 九、多智能体协作机制

项目采用“受控自主协商”，而不是完全自由讨论。

### 1. 学习路径生成流程

学生请求
→ 协调器读取画像
→ 知识检索智能体提供知识结构
→ 教学设计智能体提出教学策略
→ 路径规划智能体形成初步路径
→ 评估诊断智能体检查难度适配性
→ 内容审核智能体审核
→ 协调器输出最终路径

### 2. 学习资源生成流程

学生选择知识点
→ 知识检索智能体提供知识范围
→ 教学设计智能体确定讲解方式
→ 文档与实践资源智能体生成内容
→ 题库智能体生成配套练习
→ 内容审核智能体检查准确性
→ 协调器组合为学习资源包

### 3. 错题诊断流程

学生提交答案
→ 评估诊断智能体识别错误类型
→ 知识检索智能体定位薄弱知识点
→ 教学设计智能体确定补救方式
→ 文档智能体生成补充讲解
→ 题库智能体生成变式题
→ 审核智能体检查
→ 更新学生画像

这种设计既体现智能体之间的信息交换和协商，又能控制调用成本、响应时间和结果稳定性。

## 十、个性化资源设计

### 1. 个性化知识讲解

同一知识点根据学生基础生成不同版本。

零基础版本强调：

* 通俗类比；
* 逐步解释；
* 少量术语；
* 图示和简单代码。

进阶版本强调：

* 抽象定义；
* 时间和空间复杂度；
* 多种实现比较；
* 典型应用场景。

### 2. 知识点思维导图

系统输出Mermaid思维导图或流程图代码，并在前端渲染。

例如链表知识图：

链表
→ 节点结构
→ 指针连接
→ 插入操作
→ 删除操作
→ 遍历
→ 单链表
→ 双链表
→ 循环链表

### 3. 分层练习题

每个知识点生成基础、应用和提升三级题目，学生可以选择题量和难度。

### 4. 错题解析

错题解析不只显示答案，还要回答：

* 错在哪里；
* 为什么会错；
* 对应哪个知识点；
* 正确思考过程是什么；
* 下一道应该练什么。

### 5. 代码实操案例

案例包括：

* 任务说明；
* 初始代码；
* 实现要求；
* 示例输入输出；
* 关键提示；
* 参考代码；
* 常见错误。

### 6. 项目式学习任务

围绕多个知识点设计小型项目，例如：

* 学生成绩管理系统；
* 浏览器前进后退模拟；
* 银行排队系统；
* 校园导航最短路径；
* 通讯录快速查找；
* 图书借阅排序系统。

### 7. 学习总结卡片

每个章节结束后生成一张总结卡片，包括：

* 本节核心概念；
* 必须掌握的操作；
* 复杂度结论；
* 高频易错点；
* 一句话记忆；
* 下一节先修提示。

## 十一、系统页面设计

为降低开发难度，系统设置六个主要页面。

### 1. 登录与欢迎页

采用昵称或临时用户方式进入，不采集真实姓名、学号和手机号。

### 2. 对话画像页

学生通过对话说明自身情况，系统生成九维画像卡片，并允许修改。

### 3. 学习驾驶舱

展示：

* 当前课程进度；
* 当前学习模块；
* 今日学习任务；
* 学习路径；
* 易错知识点；
* 推荐资源。

### 4. 个性化资源中心

提供七类资源生成入口：

* 知识讲解；
* 思维导图；
* 分层题库；
* 错题解析；
* 代码案例；
* 项目任务；
* 总结卡片。

### 5. 学习与练习页

学生阅读讲解、查看思维导图、完成题目并提交答案。

### 6. 学习诊断页

展示：

* 当前掌握情况；
* 最近错误类型；
* 薄弱知识点；
* 推荐复习内容；
* 下一阶段学习建议。

不设置教师端，不开发复杂后台管理系统。

## 十二、推荐技术方案

考虑单人开发和一天内完成初赛，采用以下轻量技术栈。

### 1. 应用框架

**Python＋Streamlit**

选择理由：

* 不需要分别开发前端和后端；
* 支持对话框、卡片、进度条和Markdown；
* 能快速渲染Mermaid或HTML；
* 部署和运行方式简单；
* 适合制作比赛演示原型。

### 2. 大模型接口

采用OpenAI兼容接口封装，大模型服务地址、模型名称和密钥通过环境变量配置。

配置示例：

* `LLM_BASE_URL`
* `LLM_API_KEY`
* `LLM_MODEL`

后续切换科大讯飞兼容接口时，不需要大幅修改业务代码。

### 3. 多智能体实现

初赛不强制引入LangGraph、CrewAI等重型框架，采用自定义Python智能体类实现：

* `BaseAgent`
* `KnowledgeAgent`
* `TeachingAgent`
* `ResourceAgent`
* `QuestionAgent`
* `PathAgent`
* `EvaluationAgent`
* `SafetyAgent`
* `Coordinator`

各智能体通过统一JSON格式传递消息，并将交互日志保存在本地。

### 4. 数据存储

采用SQLite保存：

* 用户画像；
* 学习路径；
* 学习记录；
* 答题记录；
* 错题信息；
* 资源生成记录。

课程知识库采用JSON或Markdown文件保存。

### 5. 思维导图

使用Mermaid文本生成并在页面渲染，不依赖付费图像生成接口。

### 6. 演示保障

系统提供两种运行模式：

* **在线生成模式**：调用配置的大模型API；
* **演示缓存模式**：在API不可用时读取预置示例结果。

演示缓存模式只能作为比赛现场稳定性保障，文档中需如实说明，不能将缓存内容描述为实时生成。

## 十三、知识库设计

虽然不采用完整RAG，系统仍需建设结构化课程知识库。

每个知识点包含：

```json
{
  "chapter": "线性表",
  "topic": "单链表插入",
  "prerequisites": ["指针基础", "结构体"],
  "difficulty": 2,
  "concepts": ["节点", "头指针", "前驱节点"],
  "common_errors": ["丢失后继节点", "空指针访问"],
  "keywords": ["linked list", "insert", "node"],
  "source_file": "chapter02_linked_list.md"
}
```

知识检索采用：

* 章节匹配；
* 关键词匹配；
* 先修关系匹配；
* 难度匹配；
* 易错点匹配。

系统不向学生显示正式学术引用，但可显示“课程知识库章节来源”，增强可信度。

## 十四、安全与防幻觉机制

### 1. 内容范围限制

所有课程内容限定在结构化数据结构知识库及课程大纲范围内。

### 2. 系统提示词保护

用户输入与系统提示词分离，不允许用户指令覆盖智能体角色和审核规则。

### 3. 输入过滤

检测并阻断：

* 要求泄露系统提示词；
* 与课程无关的危险内容；
* 代码注入式提示词；
* 超长异常输入；
* 敏感个人信息。

### 4. 输出审核

内容审核智能体对生成结果进行事实一致性、答案一致性和代码合理性检查。

### 5. 代码安全

初赛版本不在服务器直接执行学生提交的任意代码，避免恶意代码运行风险。代码案例仅进行文本分析和规则检查。

### 6. 隐私保护

* 不要求填写真实姓名和学号；
* 用户昵称可匿名；
* API密钥不写入源代码；
* 日志中隐藏密钥和个人信息；
* 提交材料中不包含真实用户数据。

### 7. 数据脱敏

对手机号、邮箱、身份证号等输入使用正则表达式进行掩码处理。

## 十五、核心创新点

### 创新点一：面向学习决策的多智能体协同

区别于单一问答机器人，系统将知识定位、教学设计、资源生成、路径规划、诊断和审核分配给不同智能体，通过受控协商生成完整学习方案。

### 创新点二：画像驱动的资源差异化生成

资源内容不是简单根据知识点生成，而是同时考虑基础、目标、认知水平、学习节奏、内容偏好和易错点，使同一知识点形成不同教学版本。

### 创新点三：基于知识先修关系的动态学习路径

系统将数据结构课程组织为具有依赖关系的知识网络。学生若缺少指针、递归或复杂度等前置基础，系统会自动插入补充任务，而不是机械按照教材章节推进。

### 创新点四：从错误答案到补救资源的闭环

系统不仅判断对错，还识别错误类型，并自动生成针对性解释和变式练习，实现“错误诊断—知识补救—再次练习”的闭环。

### 创新点五：轻量多模态资源组合

在不依赖视频生成和付费图像模型的情况下，通过文字、思维导图、代码、题卡和路径图构成低成本、多形态学习资源体系。

## 十七、演示用核心场景

### 演示用户

计算机专业大一学生，数据结构零基础，C++基础较弱，每天可学习1小时，期末目标为85分，偏好图示与代码结合的学习方式。

### 演示流程

#### 第一步：输入自然语言

“我是计算机专业大一学生，没有学过数据结构，C++指针掌握得不太好，每天能学习1小时，希望期末达到85分。我喜欢通过图和代码学习。”

#### 第二步：生成画像

系统提取：

* 专业年级：计算机大一；
* 知识基础：初级；
* 学习目标：期末85分；
* 认知水平：理解层；
* 学习节奏：常规；
* 内容偏好：图示＋代码；
* 易错知识点：指针；
* 学习投入度：每天1小时；
* 历史成绩：待补充。

#### 第三步：生成路径

系统自动增加“指针与结构体预备模块”，随后安排：

算法基础
→ 线性表
→ 栈与队列
→ 树
→ 图
→ 查找
→ 排序
→ 综合项目

#### 第四步：生成链表资源包

展示：

* 零基础知识讲解；
* 链表思维导图；
* C++代码案例；
* 三级练习题；
* 学习总结卡片。

#### 第五步：展示错题诊断

学生提交一段存在空指针问题的代码，系统识别为“边界条件与空指针错误”，生成针对性解释和一道变式练习。

#### 第六步：更新学习建议

系统将“链表边界处理”加入易错知识点，并建议完成两道基础题后再学习双链表。

## 二十、项目方案结论

智构学伴以数据结构零基础学生为核心对象，以对话式动态画像为起点，通过七个专业智能体完成知识检索、教学设计、资源生成、题库建设、学习路径规划、错误诊断和内容审核。

在单人、零预算和短周期限制下，项目采用Streamlit、SQLite、结构化课程知识库和兼容式大模型接口，避免开发复杂前后端、视频生成、完整RAG和代码沙箱，将有限时间集中于评分占比最高的多智能体功能、个性化资源和可运行演示。

该方案能够满足赛题基础功能要求，并通过知识先修关系、错误补救闭环和受控多智能体协商形成较明确的创新差异。
